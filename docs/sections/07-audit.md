# 第三部分 · Agent在审计中的应用（章节10—13）

- **主线时长：** 25分钟；完整运行两项互动约35分钟
- **定位：** 不再只讲抽象“审计智能体”，而是用智能问数、智能报告生成等具体功能说明如何建设
- **总路线：** 功能全景 → 智能问数 → 智能报告 → 共性底座与治理

## 10 · 先定义审计功能（5分钟，`#audit`）

### 六类功能

1. 智能问数：自然语言问题→可信数字、口径、查询、来源和限制。
2. 智能风险筛查：规则/模型→疑点队列、触发原因和风险信号。
3. 智能取证调查：Agent按证据缺口调用只读工具→证据包。
4. 智能底稿助手：已复核证据+程序模板→带引用的底稿草稿。
5. 智能报告生成：冻结发现库+回应+模板→受控报告草稿。
6. 持续审计监控：交易事件+规则/模型→预警、任务和关闭记录。

### 教学重点

- 一个聊天框不等于一个审计平台。
- 问数、筛查、取证和报告需要不同输入、控制与工作产品。
- 首个功能应边界清楚、数据可得、输出可评价且有人复核。

## 11 · 智能问数（7分钟，`#audit-architecture`）

### 定义与交付

智能问数让审计人员用自然语言提问，但系统只在授权和认证口径内查询。可信回答必须包含：

1. 数字与解释（Answer）
2. 指标、范围和比较口径（Definition）
3. 可审阅查询或语义计划（Query）
4. 数据快照、表字段和血缘（Source）
5. 缺失、权限和适用限制（Caveat）

### 七步流程

问题受理 → 口径契约 → 元数据目录/语义层 → 权限检查 → 安全查询 → 结果校验 → 证据化回答。

### 四层架构

1. 交互层：问题、澄清、追问、结果、导出。
2. 理解与编排层：LLM识别意图、补齐契约、生成语义计划；不直接持有数据库权限。
3. 可信控制层：认证指标语义层、权限引擎、SQL验证器、结果校验器。
4. 工具数据层：元数据目录、只读查询网关、数据仓库和查询血缘日志。

### `AskDataLab`四种分支

- 指标查询：口径完整→验证后回答。
- 多表追问：继承已确认上下文→连接组织历史与规则结果。
- 口径不清：不猜测→请求澄清，暂不访问数据。
- 越权请求：权限层阻断→不生成SQL，并留下安全日志。

### 技术边界

LLM负责语言意图、澄清、语义计划和解释；代码系统负责权限、认证口径、SQL安全、执行、计算、校验与日志。

核心结论：**LLM不直接“算出”审计数字；可信问数是“自然语言→口径契约→受控查询→结果校验→数字+来源”。**

## 12 · 智能报告生成（7分钟，`#audit-evidence`）

### 受控输入

- 审计范围与方法
- 已确认、已冻结的结构化发现
- 证据ID、来源、版本与复核状态
- 管理层回应、责任人和整改日期
- 报告模板、语气、评级规则和审批流程

模型不得补写未确认原因、未量化影响、风险评级、管理层回应或审计结论。

### 八步流程

冻结发现 → 完整性检查 → 生成提纲 → 分节起草 → 引用绑定 → 数字/术语/评级一致性校验 → 人工审阅 → 文档渲染与归档。

### 四层架构

1. 可信输入层：发现登记库、证据对象库、制度模板、管理层回应。
2. 生成编排层：完整性门、提纲规划、LLM分节起草、引用绑定。
3. 校验审批层：数字勾稽、事实引用、术语/评级规则、人工审批流。
4. 受控输出层：草稿、图表、修订对比和正式归档件。

### `ReportGenerationLab`三种分支

- 证据完整：可生成带引用草稿，仍需逐句人工复核。
- 原因缺失：保留明确占位符，创建访谈/量化任务，不得猜测。
- 数字冲突：生成前阻断，由人确认正确来源并重新冻结版本。

### 人机边界

模型负责组织、表达和格式；审计人员负责证据评价、原因与影响确认、评级、建议、回应处理和最终签发。报告草稿只能引用底稿，不能反向成为底稿证据。

核心结论：**报告生成是基于已确认发现的受控起草，不是自动发现问题或自动签发报告。**

## 13 · 共性底座与治理（6分钟，`#audit-rollout`）

### 共性六层架构

数据/制度/模板来源 → 受控工具与安全 → 规则/模型/语义能力 → Agent与工作流编排 → 工作产品/证据 → 审计应用。

### 统一证据链

`数据快照 → 问数结果 → 证据对象 → 审计发现 → 报告段落`。每一节点都有对象ID、版本、状态和责任人，避免各功能复制并改写事实。

### 分功能评价

- 智能问数：问题澄清、口径选择、SQL安全、数字复算、来源可追溯和越权阻断。
- 智能报告：冻结发现忠实度、事实引用、数字图表一致、缺失保留、人工修改和误签发事件。
- 共同经营：时长、成本、节省工时、接管率、稳定性、安全及漂移。

### 上线与责任

离线基准 → 影子运行 → 小范围试点 → 受控扩围。首期MVP明确拆成：

- 问数MVP：一个主题、有限认证指标和维度、汇总查询、澄清/追问、来源展示；不开放高敏明细、任意SQL或写回。
- 报告MVP：一种模板、只读已复核发现、完整性/引用/数字校验、修订对比和Word草稿；不做自动发现、原因、评级或签发。

审计、产品流程、数据、技术模型、安全合规五方角色共同负责。

## 自动化边界

| 可以自动或受控执行 | 必须由审计人员承担 |
|---|---|
| 指标匹配、只读查询、计算与勾稽 | 确认特殊口径和解释重大波动 |
| 将已确认发现组织为草稿并绑定引用 | 评价证据、确认原因影响和风险评级 |
| 模板渲染、数字一致性和引用完整性校验 | 批准底稿、处理回应并签发报告 |
