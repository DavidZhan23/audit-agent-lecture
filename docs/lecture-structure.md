# 整堂课结构总览

**最后核对代码：** `app/page.tsx` → `export default function Home`
**课程标题：** LLM 与 Agent：基础、架构及审计应用

## 三层总体架构

1. **第一部分：大模型和智能体的技术基础**——规则 → ML → ANN → LLM，解释技术为什么逐层出现。
2. **第二部分：Agent基础与架构**——从LLM的行动缺口出发，讲清Agent的定义、架构、工具反馈、状态、控制和评价。
3. **第三部分：Agent在审计中的应用**——把通用能力组合成受控、可复核、可评价的审计智能体。

三部分不是三个并列术语清单：第一部分建立技术底座，第二部分搭出通用运行系统，第三部分用审计交付、证据和责任边界约束该系统。

## 第一部分：技术基础（`#part-1`）

开场 `#problem` 先展示三部分地图。`#code`—`#llm` 用递进案例解释上一种方法的能力边界：

| 技术 | 事项 | 一句话问题 | 为什么升级 |
|---|---|---|---|
| 规则 | BX-41610 / BX-41902 | 同一张发票报销两次 | 条件可完整表达，规则最合适 |
| ML | BX-41881—84 | 四笔均低于2,000元但集中发生 | 多个弱信号难以穷举 |
| ANN | BX-42306 | 票面286元、平台86元 | 异常藏在高维像素中 |
| LLM | BX-42519 | 客户招待与儿童餐、生日蛋糕、无拜访记录矛盾 | 需要理解语言与业务语境 |

人脸识别保留为ANN趣味支线，不替代审计主线。

## 第二部分：Agent基础与架构（`#part-2` / `#agent`）

本章按七步展开：

1. 为什么还需要Agent：LLM提出行动，不等于系统已经行动。
2. Agent究竟是什么：区分普通程序、工作流、LLM应用和Agent。
3. 系统由什么组成：目标与策略、LLM决策核、工具、状态与记忆、编排循环、控制与可观测。
4. 工具怎样被调用：工具说明、Schema、参数校验、权限、结构化返回和错误语义。
5. 反馈怎样改变下一步：状态、长期记忆、运行轨迹、动态分支和五类停止条件。
6. 怎样防止失控：建议/只读调查/受控执行三种自主度，以及最小权限、预算和人工关口。
7. 怎样评价价值：任务结果、过程质量、成本、安全和常见失败模式。

BX-42017仅用于验证动态反馈：航班结果不同会使Agent缩短调查、扩展调查或因工具失败转人工。

## 第三部分：Agent在审计中的应用（`#part-3` / `#audit`）

本章按八步展开：

1. 定义审计交付：事实、适用标准、原始证据、不确定性和核查建议。
2. 选择窄场景：用五维诊断区分适合Agent、只需规则、不应自动化和目标过宽的任务。
3. 组合技术能力：规则、ML、ANN、LLM和Agent各做擅长的部分。
4. 连接数据与制度：九张表、两份制度、关联键、版本、权限和来源。
5. 设计端到端流程：任务受理 → 规则筛查 → 模型增强 → Agent调查 → 证据包 → 人工复核。
6. 形成证据包：以BX-42017逐项检查五字段交付是否可追溯。
7. 建立治理控制：提示注入、隐私、越权、幻觉、职责和版本漂移。
8. 评价并逐步上线：离线基准 → 影子运行 → 小范围试点 → 受控扩围。

统一边界：系统可以自动筛查、受控调查和整理证据，不自动认定违规、舞弊或签发审计意见。
