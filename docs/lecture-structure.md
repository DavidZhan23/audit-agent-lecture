# 整堂课结构总览

**最后核对代码：** `app/page.tsx` → `export default function Home`
**课程标题：** LLM 与 Agent：基础、架构及审计应用

## 三层总体架构

1. **第一部分：大模型和智能体的技术基础**——规则 → ML → ANN → LLM，解释技术为什么逐层出现。
2. **第二部分：Agent基础与架构**——从LLM的行动缺口出发，讲清Agent的定义、架构、工具反馈、状态、控制和评价。
3. **第三部分：Agent在审计中的应用**——把通用能力组合成受控、可复核、可评价的审计智能体。

三部分不是三个并列术语清单：第一部分建立技术底座，第二部分搭出通用运行系统，第三部分用审计交付、证据和责任边界约束该系统。

## 第一部分：技术基础（`#part-1`）

开场 `#problem` 先展示三部分地图。`#code`—`#llm` 用递进案例解释上一种方法的能力边界：

| 技术 | 事项 | 一句话问题 | 为什么升级 |
|---|---|---|---|
| 规则 | BX-41610 / BX-41902 | 同一张发票报销两次 | 条件可完整表达，规则最合适 |
| ML | BX-41881—84 | 四笔均低于2,000元但集中发生 | 多个弱信号难以穷举 |
| ANN | BX-42306 | 票面286元、平台86元 | 异常藏在高维像素中 |
| LLM | BX-42519 | 客户招待与儿童餐、生日蛋糕、无拜访记录矛盾 | 需要理解语言与业务语境 |

人脸识别保留为ANN趣味支线，不替代审计主线。

## 第二部分：Agent基础与架构（`#part-2`，章节06—09）

不再用一个大section承载全部内容，而是拆为四个独立二级章：

1. **06 Agent：定义与边界（`#agent`）：** LLM行动缺口、Agent定义、程序/工作流/LLM/Agent区分、四问适配判定。
2. **07 Agent运行架构（`#agent-architecture`）：** 六块架构、工具契约、模型/编排器/工具责任、状态记忆轨迹、动态分支、单Agent与多Agent。
3. **08 可靠性与控制（`#agent-control`）：** 目标/模型/工具/状态/控制五类风险，自主度三档，运行前中后控制和事故处置闭环。
4. **09 价值、评价与建设（`#agent-evaluation`）：** 五类业务价值，结果/过程/成本/风险四维评价，最小原型到持续运营六阶段。

BX-42017仅用于验证动态反馈：航班结果不同会使Agent缩短调查、扩展调查或因工具失败转人工。

## 第三部分：Agent在审计中的应用（`#part-3`，章节10—13）

同样拆为四个独立二级章：

1. **10 审计任务与场景（`#audit`）：** 五要素任务契约、五字段疑点证据包、四类场景适配诊断与设计画布。
2. **11 审计智能体架构（`#audit-architecture`）：** 五种技术组合、九表两制度、六层系统架构和六环节端到端Python流程。
3. **12 证据与人机协作（`#audit-evidence`）：** BX-42017证据包、六环节人机责任、机器草稿到正式底稿的版本链与复现清单。
4. **13 治理、评价与试点（`#audit-rollout`）：** 六类治理风险、四阶段上线门槛、审计/产品/数据/技术/安全五方责任。

统一边界：系统可以自动筛查、受控调查和整理证据，不自动认定违规、舞弊或签发审计意见。
